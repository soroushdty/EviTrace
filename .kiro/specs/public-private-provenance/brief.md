# Brief: public-private-provenance

## Problem
Institutional users cannot share an evidence trace at all today: publishing provenance for a PHI-bearing or institutionally restricted corpus means publishing the anchors that point straight back into the protected text. The result is all-or-nothing — either full disclosure or no traceability. Reviewers need a public trace that proves a claim was grounded, states plainly where evidence exists but is restricted (prov R7.4, priv R12.4), and can be integrity-checked by someone outside the runtime (prov R10.4).

## Current State
- Nothing in the repo distinguishes internal from shareable output. `outputs/<paper>.extracted.json`, `outputs/qc_report.csv`, and `generate_w3c_jsonld()` in `src/artifact_generation/w3c_annotation.py` all emit a single view; `project()` reads `unified.semantic` and `unified.alignment` with no privacy filter in the path.
- Evidence anchors are raw and unconditional: `src/pipeline/evidence_index.py` items carry `text`, `page`, `coords`, `xpath`, and `source_pdf`, and the disk cache is keyed `{paper_id}_{pdf_hash}` in the clear.
- Hashing exists only for cache invalidation — `ManifestIdentity` in `src/pipeline/manifest.py` (`pdf_content_hash`, `config_hash`, `extraction_map_hash`). There is no signing, no ordered append-only history, no post-generation modification detection, no commitment scheme.
- No vault, no pseudonym mapping store, no access logging anywhere in `src/`. No `src/provenance/` or `src/privacy/` package exists yet.

## Desired Outcome
One provenance graph yields two coherent views: a private view that authorized users can resolve back to original source anchors (prov R7.3), and a public view whose every omission is explicit, whose references to restricted evidence are safe references or commitments (prov R12.3, priv R12.2), and which carries enough integrity metadata for external verification. Private↔public linkage is auditable without exposing PHI (priv R12.5), and every proof artifact states what it does and does not prove (prov R10.5, priv R13.5).

## Approach
View projection plus a commitment layer, built strictly on top of the existing graph. The public view is generated by projecting the core graph through privacy-supplied labels and disclosure decisions — provenance never re-derives sensitivity, it consults it (the prov R7.5 ↔ priv R1.3 mutual recursion resolved as **privacy decides, provenance consumes**, with the interface already pinned in `provenance-core`). The vault holds pseudonym mappings and raw anchors behind an access-logged boundary; integrity markers start at content fingerprints and ordered append-only history, with the structure left open for later proof systems (priv R13.4) rather than committing to one now.

## Scope
- **In**: prov R7 (private vs public views, traceability preserved without content exposure, authorized anchor resolution, explicit omission); prov R10 (content fingerprints, ordering and integrity checks over run history, post-hoc modification detection, external-verification metadata, integrity-vs-privacy claim separation); prov R12 (privacy labels on provenance nodes, consulting privacy metadata before public inclusion, safe-reference/commitment/omission markers, private↔public relationship preservation, disagreement → requires-review).
- **In**: priv R10 (private identity/evidence vault, mapping storage, safe references in public provenance, logged resolution access, unauthorized inaccessibility, exported references that reveal nothing alone); priv R12 (field-level public/private determination, replacement/omission/encryption/commitment, safe link from public claim to private tree, restricted-evidence-exists marker, auditable linkage); priv R13 (commitment-based references, private/public root recording with disclosure-transaction metadata, transaction signing, forward-compatible structure, explicit proof semantics).
- **Out**: PHI detection, sensitivity classification, policy evaluation, redaction/pseudonymization *transforms* themselves; the run artifact and export serializers; graph construction and chain validation.

## Boundary Candidates
- View projection (public generation from labels) vs. the vault (private storage and authorized resolution).
- Safe-reference construction vs. commitment/cryptographic integrity marking.
- Per-artifact fingerprinting vs. ordered append-only run history.
- Disclosure-transaction recording (private root, public root, metadata) vs. the disclosure decision that authorizes it (privacy-core).

## Out of Boundary
- Any legal or compliance conclusion. Integrity verification is explicitly not a privacy or compliance guarantee (prov R10.5); the anti-overclaiming requirements (priv R16) belong to privacy-core and are inherited, not restated.
- Authoring policy profiles, PHI classifiers, or leakage-risk estimation (privacy-core, privacy-transformations).
- Producing zero-knowledge proofs. Only the structural affordance for future proof integration is in scope (priv R13.4).
- Access control / identity provider integration beyond recording that an authorization check occurred and logging the access event.

## Upstream / Downstream
- **Upstream**: provenance-core (graph, node identity, pinned privacy-label interface — required); privacy-core (labels, disclosure decisions, policy gate, secret/key management for signing — required); provenance-audit-export (the artifacts being projected); `src/pipeline/evidence_index.py`, `src/pipeline/manifest.py`, `src/artifact_generation/w3c_annotation.py`.
- **Downstream**: reviewer-ui (renders public vs private status side by side, priv R15.5); external verifiers of exported traces; any future proof-system spec.

## Existing Spec Touchpoints
- **Extends**: xtrace-toolkit — `R-GOV-1`'s append-only ledger is the natural carrier for prov R10.2 ordering/integrity checks and must remain a projection of the core graph, not a second store; `R-X-2` per-artifact hashes must not be re-implemented as a separate fingerprint scheme.
- **Adjacent**: privacy-transformations (priv R6–R8 produce the transformed artifacts this spec commits to and links); provenance-audit-export (owns artifact contents and versioning; this spec redacts and signs them); privacy-core (owns priv R11 privacy audit trail — assigned to `privacy-core` by both `.kiro/steering/roadmap.md` and `.kiro/specs/privacy-core/brief.md`).
- **Note**: tamper-evidence (prov R10) is the intended home for `merckle tree.md`, a parked, unread idea document. It stays unread until this spec's design phase and must not be assumed to constrain the approach.

## Constraints
- Python 3.12.x, `src/`-layout; `src/provenance/` and `src/privacy/` dependency directions must be declared and enforced in `tests/test_dependency_directions.py` before any cross-package import — provenance depending on privacy for decisions is one-directional and must be encoded there.
- Cryptographic libraries stay lazily imported inside function bodies; no library or proof system is prescribed by requirements.
- Signing/encryption keys must be obtained only through privacy-core's managed access path (priv R2.1) and never appear in logs, prompts, provenance artifacts, or exports (priv R2.2).
- Vault storage must not land in the repo, in `outputs/`, or in any test fixture; no PHI, credentials, or real patient data anywhere.
- New top-level YAML keys registered in `_ALL_KNOWN_TOP_LEVEL_KEYS` (`src/utils/config_utils.py`).
