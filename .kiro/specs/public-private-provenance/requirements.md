# Requirements Document

## Project Description (Input)
Institutional users cannot share an evidence trace at all today. Publishing provenance for a corpus that carries protected health information or institutionally restricted material means publishing the anchors that point straight back into the protected text, so the only available choices are full disclosure or no traceability. Nothing in the repository distinguishes an internal artifact from a shareable one, evidence anchors are raw and unconditional, hashing exists only for cache invalidation, and there is no vault, no pseudonym mapping store, and no access logging anywhere.

`public-private-provenance` makes one provenance graph yield two coherent views. The private view retains enough detail for an authorized user to resolve evidence back to original source anchors. The public view is generated by consulting the disclosure decisions that the privacy subsystem already produced — provenance never re-derives sensitivity, it consults it — and every omission in it is explicit, every reference to restricted evidence is a safe reference or a commitment rather than a raw identifier, and the artifact states plainly that restricted evidence exists. A vault holds pseudonym mappings, reference salts, commitment openings, and raw anchors behind an access-logged boundary. Integrity markers begin at content fingerprints and an ordered, append-only run history whose structure is left open for later proof systems rather than committing to one now. Every proof artifact states what it proves and, just as explicitly, what it does not.

## Introduction

This spec is the disclosure-view layer over the provenance graph. It adds no new sensitivity judgement and no new decision authority: `privacy-core` decides, `provenance-core` carries the decision on every node, and this spec **consults** that carrier to project a public view. The one-directional rule — privacy decides, provenance consumes — holds throughout; no requirement below permits this subsystem to classify content, evaluate a policy, or widen a disclosure that the privacy subsystem did not authorize.

Three seams with adjacent specs are settled here rather than left ambiguous.

First, `provenance-audit-export` already ships a fail-closed, `restricted`-by-default **run-level sharing marking** driven by a configured decision-to-level map, and it treats that marking as a lookup rather than a judgement. This spec does not duplicate, replace, or contradict that. It consumes the same configured decision-to-level mapping as its input and adds only the *per-node disposition* that a level implies for public-view construction. The mapping from a privacy decision to a sharing level is defined exactly once, upstream.

Second, provenance R12.1 (privacy labels attached to provenance nodes) is deliberately split. `provenance-core` owns the carrier field on the node model and stores opaque strings in it. `privacy-core` populates it. This spec owns only the **consultation logic** — reading the carrier and deriving a disposition from it — and claims no part of the carrier's definition or population.

Third, integrity work in this spec is bounded by what it can honestly deliver. Content fingerprints, an ordered append-only hash chain over run history, post-hoc modification detection, and salted-hash commitments are specified because they are well understood and testable. Inclusion proofs, zero-knowledge attestations, at-rest encryption of the vault, and any externally verifiable authorship claim are **not** specified; the structure is required to remain forward-compatible with them, and every artifact is required to state which of these it does and does not provide.

Requirement coverage relative to the archived source documents: prov R7 in full; prov R10 in full; prov R12 in full except R12.1's carrier half, which `provenance-core` owns; priv R10, R12, and R13 in full. The anti-overclaiming obligations of priv R11 and priv R16 are **inherited** from `privacy-core`, not restated: this spec embeds the upstream disclaimer rather than authoring a second one.

## Boundary Context

- **In scope**: Generation of a public provenance view and a private provenance view from one graph; consultation of the privacy carrier to derive a per-node disposition; explicit omission markers; safe references and commitments that stand in for restricted evidence; a restricted-evidence-exists indicator on the public artifact; an auditable private-to-public linkage; a private vault holding pseudonym mappings, reference salts, commitment openings, and raw anchors, with a logged resolution path and inaccessibility without authorization; per-artifact content fingerprints and an ordered append-only integrity history with post-hoc modification detection; verification metadata and a verification procedure usable outside the runtime; disclosure transaction records carrying the private root, the public root, and disclosure metadata, with optional integrity marking of the transaction; a forward-compatible slot for future proof systems; and an explicit statement, on every integrity artifact, of what it proves and what it does not.
- **Out of scope**: Detection of protected health information, sensitivity classification, disclosure policy evaluation, and any decision about whether content may be disclosed — all owned by `privacy-core`. Redaction, pseudonymization, minimization, date-shifting, and leakage-risk estimation transforms — owned by `privacy-transformations`. Graph construction, node identity, chain validation, and completeness computation — owned by `provenance-core`. Run artifact contents, audit-package contents, interoperable export formats, fidelity reporting, and the run-level sharing marking — owned by `provenance-audit-export`. Authoring policy profiles. Producing zero-knowledge proofs. Identity-provider or access-control integration beyond recording that an authorization check occurred and logging its outcome. Any legal, regulatory, or compliance conclusion of any kind.
- **Adjacent expectations**: This subsystem expects the privacy subsystem to have supplied a disclosure decision on the nodes it governs, and treats an absent decision as restricted rather than as permission. It expects the provenance subsystem to supply the graph, the node identities, the ordered event-log projection, and the carrier structure, and it recomputes none of them. It expects the audit-export subsystem to own the decision-to-level mapping, the per-artifact file digests, and the run-level sharing marking, and it consumes all three rather than producing competing versions. It expects signing key material, where used, to be reached only through the privacy subsystem's managed secret path, and never to appear in any artifact this spec writes. It expects the operator to supply the vault location; it does not choose one. It does not own, and must not absorb, the behavior of any of those adjacent components.
- **Standing product boundaries**: This spec does not admit autonomous systematic review generation, automated clinical recommendations, guaranteed extraction correctness without human validation, scanned-document workflows beyond fallback support, or meta-analysis automation.

## Requirements

### Requirement 1: Public and Private Provenance Views

**Objective:** As an institutional reviewer, I want private provenance distinguished from shareable provenance, so that traceability can be preserved without unnecessary disclosure of sensitive source material.

#### Acceptance Criteria

1. When provenance artifacts are finalized for a run, the disclosure subsystem shall support producing a private provenance view and a public provenance view from the same underlying graph.
2. When a public provenance view is generated, the disclosure subsystem shall preserve the claim-to-evidence-to-source relationships of the underlying graph without including private evidence content that the disclosure decision does not permit.
3. When a private provenance view is generated, the disclosure subsystem shall retain sufficient detail for an authorized user to resolve each evidence reference back to its original source anchors.
4. If the public provenance view omits any element present in the private view, then the disclosure subsystem shall represent that omission explicitly in the public view rather than silently dropping the element.
5. When a public provenance view is generated, the disclosure subsystem shall record the count of elements included, reduced to a safe reference, reduced to a commitment, and omitted.
6. The disclosure subsystem shall not modify the underlying provenance graph while generating either view.
7. Where the disclosure subsystem is disabled by configuration, the workflow shall produce no public view and shall record that no disclosure view was generated rather than reporting the run as shareable.

### Requirement 2: Privacy Consultation and Node Disposition

**Objective:** As a privacy-conscious investigator, I want public-view construction to consult the disclosure decisions the privacy subsystem already made, so that traceability never accidentally discloses sensitive data and provenance never makes a sensitivity judgement of its own.

#### Acceptance Criteria

1. When public provenance is requested, the disclosure subsystem shall consult the privacy metadata carried on each provenance node before including any of that node's details.
2. When a node carries a disclosure decision, the disclosure subsystem shall resolve that decision to a sharing level using the mapping already configured for the run, and shall not define a second mapping.
3. When a sharing level is resolved for a node, the disclosure subsystem shall derive that node's public disposition from a declared level-to-disposition table drawn from the set comprising include-in-full, safe reference, commitment, omission, and requires-review.
4. If no disclosure decision has been supplied for a node, or the supplied decision is absent from the configured mapping, then the disclosure subsystem shall apply the most restrictive disposition and shall record that no usable decision was available.
5. The disclosure subsystem shall not read, interpret, compare, or branch on the sensitivity label carried alongside the decision.
6. The disclosure subsystem shall not classify content sensitivity, evaluate a disclosure policy, or produce a disclosure decision.
7. If privacy metadata and provenance records disagree for a node, then the disclosure subsystem shall assign that node the requires-review disposition, shall record the specific disagreement, and shall mark the run as requiring review.
8. When a node's disposition is derived, the disclosure subsystem shall record which component supplied the underlying decision and when it was supplied.

### Requirement 3: Safe References, Commitments, and Restricted-Evidence Markers

**Objective:** As a provenance reviewer, I want restricted evidence represented in public artifacts by references that reveal nothing on their own, so that a public trace remains useful without disclosing what it stands for.

#### Acceptance Criteria

1. If privacy metadata prevents including an evidence node's details in the public view, then the disclosure subsystem shall represent that node with a safe reference, a commitment, or an omission marker.
2. When a safe reference is produced for a node, the disclosure subsystem shall produce a value that cannot be resolved back to the underlying identifier or content without access to material held in the private vault.
3. When a commitment is produced for restricted content, the disclosure subsystem shall produce a value that binds to that exact content and shall retain, in the private vault, the material needed to later open the commitment.
4. When a public artifact contains a safe reference or a commitment in place of evidence, the disclosure subsystem shall indicate on that artifact that evidence exists but is restricted.
5. When a safe reference or a commitment is exported, it shall not reveal the underlying protected content, the underlying identifier, or the source location by itself.
6. When the same node is represented across repeated generations of the public view within one run, the disclosure subsystem shall produce the same safe reference and the same commitment value for it.
7. If a commitment cannot be produced for a node whose disposition requires one, then the disclosure subsystem shall fall back to an omission marker and shall record the reason rather than including the content.

### Requirement 4: Private-to-Public Linkage and Auditability

**Objective:** As a provenance reviewer, I want the relationship between the private trace and the public trace to be auditable, so that a public claim can be tied back to its private evidence by an authorized reviewer without exposing protected content to anyone else.

#### Acceptance Criteria

1. When a public claim references private evidence, the disclosure subsystem shall preserve a link from that public claim to the corresponding entry in the private provenance view.
2. When a privacy transformation has been applied to evidence, the disclosure subsystem shall preserve the relationship between the private evidence and its public representation as permitted by the recorded decision.
3. When private and public provenance artifacts are linked, the disclosure subsystem shall make the linkage auditable without exposing protected content in the linkage record itself.
4. When a linkage record is produced, it shall identify the public element, the disposition applied, and the decision that authorized the disposition.
5. If a public element cannot be linked back to a private element, then the disclosure subsystem shall record the unlinked element and the reason rather than emitting the public element without a linkage entry.
6. The linkage record shall contain no evidence text, no source anchor, and no raw private identifier.

### Requirement 5: Private Identity and Evidence Vault

**Objective:** As an institutional user, I want private identity mappings and raw evidence anchors held in a protected vault, so that public artifacts can preserve traceability without exposing protected content.

#### Acceptance Criteria

1. When a pseudonymous or safe identifier is created, the disclosure subsystem shall store the mapping to the underlying identifier only in the private vault.
2. When commitment openings, reference salts, or raw source anchors are retained, the disclosure subsystem shall store them only in the private vault.
3. When an authorized user resolves a private reference or anchor, the disclosure subsystem shall record an access event identifying the reference, the requesting operator, the time, and the outcome.
4. If a requester is not authorized, then the disclosure subsystem shall not return the private mapping or the raw anchor, shall record the denied access event, and shall report the denial.
5. When a vault reference appears in an exported artifact, it shall not reveal the underlying protected content by itself.
6. The disclosure subsystem shall refuse to place vault storage inside the repository or inside the run output directory, and shall report the refusal rather than falling back to a default location.
7. If no vault location has been configured, then the disclosure subsystem shall not generate safe references or commitments that would require one, and shall report the missing configuration.
8. The disclosure subsystem shall not assert that vault contents are protected by any control it does not itself provide.

### Requirement 6: Content Fingerprints and Ordered Integrity History

**Objective:** As a compliance-aware system designer, I want provenance records to carry integrity markers and an ordered history, so that an exported trace can be checked for modification after it was generated.

#### Acceptance Criteria

1. When provenance artifacts are finalized, the disclosure subsystem shall record a content fingerprint for each artifact it covers.
2. When provenance events are appended to a run history, the disclosure subsystem shall produce an ordered history in which each entry's integrity marker depends on the preceding entry's marker.
3. If an artifact or a history entry is modified after generation, then the disclosure subsystem shall detect that the artifact or entry no longer matches its recorded integrity marker and shall report the first position at which the history diverges.
4. When the integrity history is produced, the disclosure subsystem shall derive its ordering from the ordered event view the provenance subsystem already provides and shall not maintain a second ordering.
5. When a content fingerprint or per-artifact digest already exists upstream, the disclosure subsystem shall reference it rather than computing a competing value.
6. When an integrity marker is recorded, the disclosure subsystem shall record the algorithm identifier and the canonicalization rule used to produce it.
7. The disclosure subsystem shall not offer any hash algorithm for tamper-evidence that is not collision-resistant.
8. If integrity history production fails for part of a run, then the disclosure subsystem shall report the run as unverifiable rather than as verified.

### Requirement 7: External Verification of Exported Artifacts

**Objective:** As an external evaluator, I want an exported public trace to carry enough integrity metadata to be checked outside the runtime, so that I can confirm the artifact I received is the artifact that was produced.

#### Acceptance Criteria

1. When a public provenance artifact is exported, the disclosure subsystem shall include integrity metadata sufficient to verify it without access to the runtime that produced it.
2. When integrity metadata is exported, it shall name the algorithm, the canonicalization rule, the schema identity, and the schema version needed to reproduce the verification.
3. When verification is performed against an unmodified set of artifacts, the disclosure subsystem shall report verification as successful and shall enumerate which artifacts were covered.
4. If verification is performed against a modified artifact, then the disclosure subsystem shall report verification as failed and shall identify the artifact and the position at which the mismatch was found.
5. If an artifact named by the integrity metadata is absent, then the disclosure subsystem shall report it as uncovered rather than as verified.
6. When integrity metadata declares that a check requires material not present in the export, the disclosure subsystem shall state that the check is not externally performable.

### Requirement 8: Disclosure Transaction Recording and Integrity Marking

**Objective:** As an advanced infrastructure designer, I want each act of publishing a public view recorded as a transaction tying the private root to the public root, so that the derivation of a public artifact from a private one is itself recorded and checkable.

#### Acceptance Criteria

1. When a public provenance view is derived from a private provenance view, the disclosure subsystem shall record a disclosure transaction containing the private root marker, the public root marker, and the disclosure metadata in effect.
2. When a disclosure transaction is recorded, it shall reference the disclosure decisions and policy profile identities that authorized the dispositions applied.
3. Where transaction signing is enabled, the disclosure subsystem shall integrity-mark the finalized transaction and shall record the key-version identifier of the key in effect.
4. When a transaction is integrity-marked, the disclosure subsystem shall record whether the resulting marking can be verified by a party that does not hold the signing key.
5. The disclosure subsystem shall not place key material, or any value from which key material could be derived, into a transaction record or any other exported artifact.
6. If signing is enabled but the signing key cannot be obtained through the managed secret path, then the disclosure subsystem shall not emit an unsigned transaction as though it were signed, and shall report the failure.
7. When a disclosure transaction is finalized, it shall be appended to the ordered integrity history for the run.

### Requirement 9: Forward-Compatible Proof Structure

**Objective:** As an advanced infrastructure designer, I want the artifacts to leave room for future proof systems, so that stronger attestations can be added later without invalidating artifacts produced today.

#### Acceptance Criteria

1. Where zero-knowledge-style proof support is unavailable, the disclosure subsystem shall still produce artifacts whose structure can carry such proofs later.
2. When a commitment, an integrity history entry, or a disclosure transaction is produced, it shall carry a scheme identifier and a scheme version.
3. When an artifact carries a slot for proof artifacts, an empty slot shall be a valid and explicitly reported state rather than an absent field.
4. If an artifact declares a scheme the reader does not recognize, then the disclosure subsystem shall report the unrecognized scheme rather than interpreting the artifact under a different scheme.
5. When an extension supplies additional integrity metadata, the disclosure subsystem shall preserve it in a form clearly distinguishable from the fields it defines.
6. The disclosure subsystem shall not ship a proof system and shall not represent the proof slot as evidence that one exists.

### Requirement 10: Explicit Proof Semantics and Claim Separation

**Objective:** As a responsible project maintainer, I want every integrity artifact to state plainly what it proves and what it does not, so that integrity verification is never mistaken for a privacy or compliance guarantee.

#### Acceptance Criteria

1. When cryptographic commitments or integrity markers are used, the disclosure subsystem shall distinguish integrity verification from any privacy or compliance guarantee.
2. When a proof or integrity artifact is present, it shall state what it establishes and what it does not establish.
3. When a public view, an integrity artifact, or a disclosure transaction is exported, it shall carry the anti-overclaiming statement defined by the privacy subsystem rather than a separately authored statement.
4. The disclosure subsystem shall not produce any artifact whose name, field, or value asserts that a view, a run, or an export is compliant, certified, anonymized, de-identified, or approved by any external authority.
5. When a safe reference or commitment is described, it shall be described as an integrity and disclosure-control mechanism rather than as a confidentiality guarantee for the material it stands for.
6. Where a capability is absent, the disclosure subsystem shall say so explicitly rather than leaving its absence to be inferred.

### Requirement 11: Fail-Closed View Generation

**Objective:** As a clinical AI safety reviewer, I want every failure in view generation to fail closed, so that uncertainty never results in an over-disclosing public artifact.

#### Acceptance Criteria

1. If consultation of a node's privacy metadata fails for any reason, then the disclosure subsystem shall treat that node as restricted.
2. If disposition derivation cannot reach a result for a node, then the disclosure subsystem shall omit that node's details and shall record the failure.
3. If vault access fails, is unavailable, or is denied, then the disclosure subsystem shall not include the material that access would have provided.
4. If integrity marking fails for an artifact, then the disclosure subsystem shall report that artifact as unverifiable and shall not report the run as verified.
5. The disclosure subsystem shall have no configuration setting, environment variable, or code path whose effect is to include content in a public view when a disposition could not be derived.
6. When a failure blocks one element, the disclosure subsystem shall continue processing unaffected elements and shall report each blocked element individually.
7. If public view generation fails as a whole, then the disclosure subsystem shall emit no partial public artifact and shall report the failure.
